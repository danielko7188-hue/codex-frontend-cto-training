# UX/UI Review Pipeline

Use this on every frontend project by default.

## 1. Discover

- inspect the repository
- locate public surfaces and internal surfaces
- identify the highest-leverage screens

## 2. Run

- start the app locally when feasible
- confirm the entry URLs
- collect desktop and mobile screenshots yourself

## 3. Audit

Review the current UI for:

- hierarchy
- typography
- spacing rhythm
- visual clutter
- trust and proof
- CTA clarity
- mobile quality
- product fit

## 4. Score

Use [design-scorecard.md](../../shared-references/design-scorecard.md) and score the current UI honestly.

## 5. Define the redesign

Write three things before coding:

- visual thesis
- content plan
- interaction thesis

## 6. Implement

- upgrade the visual system decisively
- keep the business goal visible
- design for mobile and desktop together

## 7. Verify

- capture after screenshots
- compare before and after
- check responsiveness
- run relevant typecheck, build, or tests

## 8. Save the lesson

Create a project review note in [project-reviews](../project-reviews) using [project-review-template.md](../templates/project-review-template.md).

If the lesson generalizes, also update:

- [founder-taste-profile.md](../../shared-references/founder-taste-profile.md)
- the relevant reference-bank entry

## Standard outputs

Every serious review should leave behind:

- before screenshots
- after screenshots
- one review note
- at least one reusable lesson
