## How we train it

1. Pick a real product or screen.
2. Audit it honestly.
3. Inspect and capture the UI directly in the browser without requiring founder screenshots.
4. Redesign and verify in the browser.
5. Study the right open-source code systems when a task needs stronger component architecture, theming, accessibility, or AI interface patterns.
6. Study official starter templates when a new product needs a stronger default skeleton or app shell.
7. Study strong DESIGN.md packs when a product would benefit from a reusable style system written in plain text for agent workflows.
8. Use the style spectrum, its reference pack, and the design direction selector when the main problem is narrow taste range or repetitive visual output.
9. Use the business-type reference pack when product-category fit matters more than aesthetic mood alone.
10. Use the screen-type reference pack when the harder question is how a specific screen should be structured.
11. Use the pattern reference pack when the weak point is a hero, proof block, pricing matrix, dashboard shell, table, lead form, or other reusable UI pattern.
12. Use the failure-pattern pack before final signoff so recurring bad outcomes get caught and corrected early.
13. Use the copy-and-messaging pack when the layout is decent but the page still sounds generic, sales-y, vague, or low-trust.
14. Use the motion-and-interaction pack when the weak point is feedback, transitions, loading, AI activity visibility, or state change clarity.
15. Check the weakness map when a result feels generic, boxy, bland, or too safe.
16. Use the AI UX playbook when the product includes agents, copilots, chat, tools, approvals, or background AI jobs.
17. Capture what worked and what the founder disliked.
18. Update the taste profile, references, and source-library when useful.

## Default operating mode

The founder should be able to say:

- "go into this folder and improve the frontend"
- "fix the UX/UI"
- "make this feel premium"

And Codex should then:

1. inspect the repository
2. find the relevant app surfaces
3. run the app locally when possible
4. capture its own screenshots
5. audit desktop and mobile
6. redesign and verify

No screenshot hand-holding should be required by default.

## New shared memory workflow

After meaningful frontend work:

1. save a review note in [project-reviews](./design-memory/project-reviews)
2. add any strong new references into [reference-bank](./design-memory/reference-bank/README.md)
3. add any strong open-source learning sources into [source-library](./design-memory/source-library/README.md) when they teach a reusable frontend skill
4. update the founder taste profile if the lesson is broad enough to matter again

## What will make it stronger fastest

- direct founder feedback on real screens
- examples of "this feels premium" vs "this feels generic"
- curated study of elite open-source UI systems and AI interface repos
- strong DESIGN.md packs that translate premium products into agent-readable style systems
- explicit training against recurring design failure modes
- a broader style-spectrum so Codex can choose different strong visual lanes on purpose
- stronger lane-specific live references so each visual lane has concrete anchors instead of vague mood words
- stronger business-type references so product category affects design choices, not just visual taste
- stronger screen-type references so homepage, pricing, dashboard, onboarding, chat, and approval flows stop being designed the same way
- stronger pattern references so heroes, proof, tables, settings, and AI surfaces stop collapsing into generic defaults
- stronger failure-pattern checks so obvious bad outputs are easier to catch before shipping
- stronger copy-and-messaging references so the words support trust, clarity, and conversion instead of weakening them
- stronger motion-and-interaction references so interfaces feel responsive, trustworthy, and alive without becoming noisy
- repeated review of AI-native product flows, tool visibility, approvals, and workflow UX
- repeated work across different product types
- saving reusable taste rules, not one-off opinions
- self-directed visual QA reps across many projects

## What not to do

- do not scrape massive random template dumps
- do not confuse strong component code with complete visual taste
- do not optimize for novelty over clarity
- do not treat one pretty screenshot as a complete design system
